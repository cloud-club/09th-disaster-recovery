# 기초 개념 정리

## ⑴ SLI / SLO / SLA란?

- **SLI (Service Level Indicator)**
    - 지금 우리가 제공하는 서비스의 품질을 사용자 관점으로 나타내는 지표
    - 보통 SLI로 가장 많이 측정하는 분야는 **가용성과 지연시간**
        
        <aside>
        🔖
        
        서비스의 가용성이란 요청 받은 작업의 성공 여부, 지연시간은 서비스가 작업을 수행하는 데 걸리는 시간을 의미
        
        </aside>
        
- **SLO (Service Level Objective)**
    - 측정된 SLI를 통해 사용자의 기대를 충족시키기 위해 서비스가 달성해야 하는 목푯값 혹은 목표 범위
    - SLO은 주기적으로 사용자 관점에서 검토 후, 필요하다면 조금씩 조정해나가는 것이 좋음
        - ex) 기존 SLO이 너무 엄격해서 지키기 어렵거나 너무 느슨해서 바꾸어야 하는 것들이 있을까?
- **SLA (Service Level Agreement)**
    - SLO를 기반으로 고객과 맺는 계약
    - SLA를 통해 명확한 메트릭 제공, 충족하지 못한 의무에 대한 상환 청구, 제공자와 비즈니스 간 파트너십 개선 등 다양한 기능을 포함
    - 또한 SLA를 통해 고객 경험 향상, 직원 경험 개선, 신뢰할 수 있는 기존 정보 출처, 생산성 및 성능 향상의 장점을 가질 수 있음
- **Error Budget**
    - SLO은 보통 100%으로 설정하지 않음. 당연하게도 모든 시스템은 완벽하지 않고 변수가 생길 수 있기 때문
    - 가용성 SLO이 98%라고 할때, 나머지 2%의 의미가 바로 이것. SLO를 충족하지 않아고 괜찮은 여유 공간을 Error Budget이라고 함
    - Error Budget을 다음과 같이 활용
        - 새로운 기능 배포
        - 시스템 업데이트
        - 장애 발생
    - 🚧 이 3%의 예산을 써버린다면
        - 신규 기능 개발 및 신규 버전 배포 중단
        - Error Budget이 소진된 Root Cause 분석 및 문제 해결

## ⑵ Metrics / Logs / Traces란?

- **Metrics**
    - **시스템의 건강 지표**, 숫자로 표현되는 집계 데이터
    - CPU 사용률, 메모리 사용량, 요청 수 (RPS), 에러율…
    - 메트릭 데이터로 ‘평소보다 응답 시간이 느려졌나?’와 같은 답을 얻을 수 있음
- **Logs**
    - **특정 사건의 상세 기록**, 이벤트 단위로 기록된 텍스트 데이터
    - 로그를 통해 ‘로그인 실패 원인’과 같은 질문의 답을 얻을 수 있음
- **Traces**
    - 요청 하나가 시스템을 지나가는 전체 경로 추적
    - 이 요청이 어떻게 처리되었는가?를 분석 가능
    - ‘결제 요청이 느린데 인증 서버의 문제인가? DB 조회의 문제인가?’와 같은 질문의 답을 얻을 수 있음

# 장애를 어떻게 대응해야 할까?

장애 대응의 1순위는 데이터에 근거한 가시성 확보라고 생각합니다. 장애가 일어났을 때, ‘이런 상황이 있었고 이런 장애였을 것이다’라는 근거없는 논리는 장애를 대응하는 자세에게 잘못되었다고 생각합니다. 시스템 내부 상황을 객관적인 지표로 확인하여 해결책과 이후 방어 체계를 구성하는 것이 옳은 장애 대응이라고 생각합니다. 따라서 모니터링 도구로 장애의 범위를 명확하게 파악하고 로그와 메트릭 데이터를 통해 원인을 찾아내는 것이 앞으로의 반복되는 실수를 막고 최단 시간 내에 복구 목표를 이룰 수 있을 것입니다.

# 장애 대응에서 가장 중요한 것은 무엇일까?

장애 대응에서 가장 중요한 것은 역시 ‘장애를 얼마나 빠르게 찾아냈는가’ 이라고 생각합니다. 장애가 일어나도 개발자들이 모른 상태로, 사용자의 cs를 통해 문제를 발견하게 된다면 그게 장애 대응 시나리오 중 최악의 상황이라고 생각합니다. 장애를 빠르게 찾아내기 위해서는 당연하게도 모니터링 데이터와 특정 장애 상황 알림 시스템이 필요하기 때문에 이 질문에 답을 하기 위해서는 모니터링 시스템을 체계적으로 구성해야 합니다. 

또한 개발자로서는 우선 일정 목표 시간 내로 장애 복구가 안될 시, 빠르게 전 버전으로 롤백하는 것이 운영 측면에서 가장 안전하다고 느꼈습니다… 가장 위험한 점이 오류를 해결하거나 범인을 찾는다고 롤백 시기를 늦춰 서비스의 품질을 떨어뜨리는 것입니다.

# 장애 대응 플로우 런북

## ⒈ 장애 시나리오

- **환경**
    - Kubernetes Cluster (단일 노드 또는 리소스 부족 상태)
    - nginx 기반 Pod (resource 설정 없음)
- **장애 현상**
    - Pod가 Evicted 상태로 종료됨
    - 서비스 일부 응답 불가

## ⒉ 장애 인지

- **`kubectl get pods`**
    
    → Pod 상태 Evicted 확인
    
- **`kubectl describe pod`**
    
    ```java
    The node was low on resource: memory
    ```
    
- **`kubectl describe node`**
    
    → MemoryPressure: True 확인
    

## ⒊ 대응 플로우

```java
1. Evicted Pod 확인
2. node 상태 확인 (MemoryPressure 여부)
3. Pod QoS Class 확인
4. 노드 리소스 사용률 확인
5. 임시 복구 (Pod 재생성)
6. 근본 원인 분석
7. 리소스 / 아키텍처 개선
```

## ⒋ 초기 대응

- Evicted Pod 삭제 및 재생성
    
    ```java
    kubectl delete pod resource-pod
    ```
    
- 필요 시 다른 노드로 스케쥴 유도
- 장애 확산 여부 확인

## ⒌ 원인 분석

- 노드 메모리 부족 상태 (MemoryPressure=True)
- 메모리 사용량
    - Requests: 87%
    - Limits: 125%
- Pod QoS Class
    - BestEffort (리소스 미설정)

⇒ 노드 메모리 부족 + BestEffort Pod 우선 eviction 발생

## ⒍ 사후 방지 대책

- 리소스 설정 의무화
    
    ```java
    resources:
      requests:
        memory: "128Mi"
      limits:
        memory: "256Mi"
    ```
    
- 노드 리소스 확장
    - Node Scale-out 또는 Scale-up
- 모니터링 강화
    - memory 사용량 기반 alert 설정
    - eviction 발생 전 알림
- QoS 개선
    - BestEffot 사용 금지
    - 주요 서비스 > Guaranteed 또는 Burstable
