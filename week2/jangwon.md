# 기초 개념 정리

## 1. SLI / SLO / SLA란?

- SLI (Service Level Indicator): 서비스의 성능을 판단하기 위해 측정하는 구체적인 척도
  - 예: 가용성(Availability), 응답 시간(Response Time), 처리량(Throughput), 에러율(Error Rate)
- SLO (Service Level Objective): SLI를 기반으로 설정한 목표 수치. 엔지니어가 지켜야하는 실질적인 목표
  - 예: 한 달 동안 API 응답 성공률을 99.9% 이상 유지
- SLA (Service Level Agreement): 서비스 제공자와 사용자 간의 비즈니스적 계약. SLO를 지키지 못했을 때의 책임, 보상, 대응 기준까지 포함
  - 예: SLO를 지키지 못했을 때 요금의 10%를 환불

| 구분 | 성격 | 질문 | 예시 |
| :--- | :--- | :--- | :--- |
| SLI | 지표 | "무엇을 측정할 것인가?" | "HTTP 응답 성공률" |
| SLO | 목표 | "어느 정도의 수치를 유지할 것인가?" | "한 달간 성공률 99.9% 유지" |
| SLA | 계약 | "목표를 못 지키면 어떻게 보상할 것인가?" | "99.9% 미달 시 요금 10% 환불" |

## 2. Metrics / Logs / Traces란?

- Metrics: 일정 시간 동안 측정된 "수치 데이터". 시스템의 상태를 숫자로 나타냄
  - Prometheus, Grafana, Thanos, Mimir
- Logs: 특정 시점에 발생한 "이벤트의 기록". 타임스탬프와 함께 텍스트로 기록됨
  - ELK Stack, Loki, Fluentd, Fluent Bit
- Traces: 하나의 요청이 분산된 시스템(마이크로서비스 등)을 거치면서 이동하는 과정을 추적한 데이터
  - Jaeger, OpenTelemetry, Tempo

## 장애를 어떻게 대응해야 할까?

장애가 발생한 원인을 파악하는 것도 중요하지만, 서비스가 빠른 시간에 정상적으로 동작하게 되돌리는 것이 더 중요하다고 생각하기 때문에 복구가 우선이라고 생각합니다.

서비스가 안정된 이후에 메트릭, 로그, 트레이스를 통해 장애가 발생한 원인이 무엇인지 파악합니다.

이렇게 파악한 원인이 추후에 재발하지 않도록 문제를 해결하고, 만약 반복되는 장애라면 자동으로 대응할 수 있도록 자동화하는 과정도 필요할 것 같습니다.

유사한 장애가 발생했을 때 참고가 될 수 있도록 문서로 정리하는 것까지가 장애 대응의 과정이라고 생각합니다.

## 장애 대응에서 가장 중요한 것은 무엇일까?

장애가 발생했을 때 빠르게 인지하기 위해 모니터링과 알림 시스템을 구축하는 것이 가장 중요하다고 생각합니다.

또한 처음 보는 장애가 발생하면 사람은 당황할 수 밖에 없기 때문에 장애 대응을 위한 SOP를 작성해두고, 트러블 슈팅에 참고하도록 하는 것도 중요한 것 같습니다.

마지막으로 자신이 작업 중 일으킨 장애가 있다면 장애를 덮거나 대응을 미루려 하지 말고, 솔직하게 인정하고 장애 상황을 공유하는 문화가 필수라고 생각합니다.

# 장애 대응 플로우 런북

## 1. 장애 시나리오

- 환경: 단일 노드 K8s 클러스터
- 장애 상황: `The node was low on resource: ephemeral-storage. Threshold quantity: 5701946799, available: 5472032Ki.` 라는 임시 저장 공간 고갈 현상

---

## 2. 장애 인지

- CI 툴로 사용하고 있는 Tekton의 파이프라인 실행이 실패했다는 알림을 받음
- 대시보드를 통해 실패한 Task의 상태를 보니 위와 같은 에러가 표시됨

---

## 3. 대응 플로우

1. `df -h` 명령어로 저장 공간 사용량 확인
2. 큰 용량을 차지하고 있는 디렉토리를 추적해 원인 분석
3. 파일 등을 삭제해 저장 공간 확보

---

## 4. 초기 대응

- `sudo crictl rmi --prune`, `sudo crictl rm --all` 등 명령어로 미사용 중인 이미지와 컨테이너를 먼저 삭제
- `/var/log` 디렉토리의 너무 큰(100MB 이상) 로그 파일들을 삭제
- `/var/lib/containers/storage/overlay` 디렉토리에서 사용하지 않는 컨테이너 추적 후 삭제

---

## 5. 원인 분석

컨테이너에서 발생한 로그가 과다하게 쌓여있었고, CI/CD로 잦은 배포로 인해 컨테이너/이미지 캐시가 쌓여서 이미지 레이어가 정리되지 않았음

---

## 6. 사후 방지 대책 🚀

- 리소스 제한 적용
  ```
  resources:
    requests:
      ephemeral-storage: "500Mi"
    limits:
      ephemeral-storage: "1Gi"
  ```

- 볼륨(스토리지) 자체를 늘려주기
