# 기초 개념 정리

## 1. SLO, SLI, SLA란?
- 모두 **서비스 수준**에 대한 용어로, 특정 기간 동안 사용자에게 제공되는 서비스 품질을 '측정 가능'하게 설명하기 위해 사용된다.
- **SLI 측정 → SLO 목표 설정 → SLA 약속**

#### (1) 서비스 수준 지표(SLI, Service Level Indicator): 서비스 품질을 측정할 수 있는 정량적 지표.
- 예시: 
    - 응답시간 (Latency): 요청~응답 소요 시간
    - 가용성 (Availability): 정상 응답 비율 또는 서비스 가동 시간 비율
    - 처리량 (Throughput): 단위 시간당 처리 요청 수
    - 내구성 (Durability): 데이터 손실 없이 보존되는 비율 (스토리지 서비스에 주로 사용)

#### (2) 서비스 수준 목표(SLO, Service Level Objective): SLI가 어떤 수준 이상이어야 한다는 팀 내부의 목표.
- SLO는 SLI를 기반으로 내부적으로 설정된 서비스 성능 수준 목표.
- 보통 가용성, 응답 시간, 처리량 등을 기준으로 설정됨.
- SLO에서 파생되는 핵심 개념이 **Error Budget**이다.
    - SLO가 99.9%이면 한 달(43,200분) 중 **43.2분**의 장애가 허용됨
    - 이 허용 여유를 Error Budget이라 하며, 배포/실험 가능 여부의 기준이 됨
    - Error Budget이 소진되면 신규 배포를 중단하고 안정화에 집중해야 한다는 신호

#### (3) 서비스 수준 계약(SLA, Service Level Agreement): 고객 또는 외부에 공식적으로 서비스 품질 수준을 약속하는 것.
- SLA는 기업과 고객 간의 법적 계약임. 미달 시 보상 포함.
- SLA는 SLO보다 보수적으로 설정하는 것이 일반적.

```
SLI ≥ SLO ≥ SLA
(실제 측정값) ≥ (내부 목표) ≥ (외부 약속)
```

#### (4) SLI, SLO, SLA 설정 예시
- SLI: API 성공률 99.95%, p95 latency 180ms ← 실제 측정값
- SLO: 성공률 99.9% 이상 유지 ← 내부 목표
- SLA: 99.5% 미만 시 크레딧 지급 ← 외부 약속



## 2. Metrics, Logs, Traces란?

- **옵저빌리티(Observability, 관측 가능성)**: 시스템 내부 상태를 외부 데이터로 추론 가능한 상태. 예상하지 못한 문제까지 분석 가능. 단순 모니터링보다 범위가 넓다. (모니터링: 정해진 지표를 보는 것 / 옵저빌리티: 예상하지 못한 문제까지 분석 가능)
    - 구성: Metrics(상태 변화), Logs(이벤트 기록), Traces(요청 흐름)
    - 흐름: Metrics로 이상 감지 → Logs로 원인 확인 → Traces로 병목 구간 특정

#### 메트릭 (Metrics): '시스템 상태'를 숫자로 집계한 시계열 데이터.
- 숫자만 저장하는 경량 데이터라 수집 비용이 낮고, 집계·통계 처리에 유리.
- 지속적으로 수집되며, 대시보드와 알람의 기반이 됨
- 예시:
    - 시스템: CPU 사용률, 메모리 사용량, 디스크 I/O
    - 애플리케이션: RPS(초당 요청 수), 에러율, p99 latency
    - 비즈니스: 결제 성공률, 활성 사용자 수
- 대표 도구: Prometheus(수집), Grafana(시각화·알람)


#### 로그 (Logs): 시스템에서 발생한 '이벤트' 기록.
- Metrics가 "이상하다"는 신호를 주면, Log를 통해 원인을 추적.
- 상세한 맥락 — 요청 파라미터, 에러 메시지, 스택 트레이스 등을 담을 수 있음.
- 비용이 높음 — 데이터 양이 많아 저장·검색 비용이 Metrics보다 큼.
- Log 레벨:
    - DEBUG: 개발 중 상세 흐름 확인용
    - INFO: 정상적인 주요 이벤트 기록
    - WARN: 잠재적 문제, 아직 에러는 아님
    - ERROR: 처리 실패, 즉각 확인 필요
    - FATAL: 시스템 중단 수준의 심각한 오류
- 대표 도구: Loki(Grafana 연동), ELK Stack(Elasticsearch + Logstash + Kibana)

#### Traces: 요청이 시스템을 통과하는 '전체 경로와 소요 시간'
- MSA 환경에서 하나의 요청이 여러 서비스를 거칠 때 병목 구간을 특정.
- 구성:
    - `Trace`: 하나의 요청 전체 흐름
    - `Span`: 각 서비스/함수 단위의 처리 구간
- 비용이 가장 높음 — 모든 요청을 수집하면 부담이 크기 때문에 일반적으로 샘플링을 적용.
    - 샘플링: 모든 트레이스를 저장하지 않고 일부만 선택해서 수집하는 방식.
        - Head-based sampling: 요청 시작 시 샘플링 여부 결정
        - Tail-based sampling: 요청이 끝난 뒤 결과를 보고 샘플링 결정 (에러, 지연 위주로 선택)
- 대표 도구: Jaeger, Zipkin, OpenTelemetry(Metrics/Logs/Traces 통합 수집 표준)

## 장애를 어떻게 대응해야 할까?

- 장애 대응은 사전에 정의된 절차에 따라 일관되게 수행되어야 함.
- 실제 장애 상황에서는 판단이 흔들리기 쉬우므로, 미리 대응 흐름을 명확히 정리해두는 것이 중요.

#### 장애 대응 기본 흐름
```
1. 인지 (Detect)      → 장애 발생 여부 확인
2. 분류 (Triage)      → 영향 범위와 심각도 판단
3. 대응 (Respond)     → 서비스 영향 최소화 조치
4. 분석 (Diagnose)    → 근본 원인 파악
5. 복구 (Resolve)     → 정상 상태로 복구
6. 회고 (Postmortem)  → 원인 정리 및 재발 방지
```

#### 인지 (Detect)
- 장애를 인지하는 경로: 모니터링 시스템(Grafana, PagerDuty 등)의 자동 알림, 내부 대시보드 확인, 사용자 문의 및 CS 티켓 등
- 가장 나쁜 상황은 **사용자로부터 먼저 장애가 인지되는 것** → SLO 기반 알람 체계를 갖추는 것이 중요.

#### 분류 (Triage)
- 장애 발생 시 가장 먼저 수행해야 할 것은 **심각도** 판단. 심각도(Severity)에 따라 대응 인원과 속도가 달라짐.
    - P1 (Critical): 전체 서비스 중단 또는 핵심 기능 불가 등 즉각적인 대응이 필요한 수준
    - P2 (High): 주요 기능 장애로 일부 사용자에게 영향이 발생하는 수준
    - P3 (Medium): 기능 일부 저하가 있으나 우회가 가능한 수준
    - P4 (Low): 서비스 영향이 거의 없는 경미한 오류 수준


#### 대응 (Respond)
- 원인 파악보다 **피해 최소화**가 우선. 원인은 서비스 복구 후 분석.
- 대표적인 초기 대응 방법:
    - Rollback: 최근 배포가 원인이라면 이전 버전으로 되돌림
    - Pod Restart: 애플리케이션 오작동 시 재시작으로 임시 복구
    - Feature Flag Off: 문제가 된 기능만 즉시 비활성화
    - Scale Out: 부하 증가가 원인이라면 인스턴스를 늘림
    - Traffic 차단: 특정 엔드포인트나 사용자 트래픽을 일시 차단

#### 분석 (Diagnose)
- 가설을 세우고 데이터로 검증하는 방식으로 범위를 좁혀 나감.
- 효율적인 접근 순서: Metrics → Logs → Traces
    - Metrics: 에러율 증가 시점과 영향 서비스 확인
    - Logs: 동일 시간대 에러 메시지 확인
    - Traces: 지연 또는 타임아웃 구간 확인

#### 회고 (Postmortem)
- "Blameless Postmortem(비난 없는 회고)". 누가 실수했는지가 아니라, 왜 그 실수가 시스템 장애로 이어질 수 있었는지 구조적 원인 분석을 진행한다..
- 회고 문서 포함 내용:
    - 타임라인 (언제 무슨 일이 있었는가)
    - 근본 원인 (5 Whys 수준의 분석)
    - 대응 과정
    - 개선 항목 & 재발 방지 대책 (Action Item, 담당자, 기한 명시)

## 장애 대응에서 가장 중요한 것은 무엇일까?

#### 1. 빠른 복구 (MTTR 최소화)

* MTTR(Mean Time To Recovery): 장애 발생부터 복구까지 평균 소요 시간.
* 장애 대응의 목표는 완벽한 원인 분석이 아니라 서비스 복구. 원인을 몰라도 롤백, 재시작, 기능 비활성화로 서비스 먼저 복구해야 함.

#### 2. 체계와 런북 (Runbook)

* 장애 상황에서는 패닉 및 실수 발생 가능성이 높으므로, 평소에 대응 절차를 미리 정리하여 문서화해두는 것이 중요.
* 런북 구성: 상황별 대응 매뉴얼.

#### 3. 커뮤니케이션

* 기술 대응만큼 상황 공유가 중요. 팀과 관련 부서가 현재 상황에 대해 동일한 정보를 보는 상태 유지.

#### 4. 회고와 개선

* 장애는 반복되므로 발생했을 때 매번 정리&학습해 시스템과 절차를 개선하는 것이 필요.
* 이 과정이 누적되며 시스템 안정성 향상.

## 장애 대응 플로우 런북

### 1. 장애 시나리오

- 배경: 사이드 프로젝트로 '독서 기록 서비스'를 개발하며 백엔드 서버 부하 테스트를 진행하던 중이었음.
- 장애 상황: 일정 수준 이상의 부하를 주자 특정 API 엔드포인트에서만 500 에러가 발생.
- 환경: Docker 컨테이너, AWS EC2 프리티어 서버
- 당시 작성했던 블로그: https://hereishyun.tistory.com/206, https://hereishyun.tistory.com/208

### 2. 장애 인지

- 부하 테스트 중 직접 응답을 확인하면서 발견. 가용성(성공률)이 떨어짐.
- 특정 구간에서 응답 지연 및 500 에러 비율 증가 확인.

### 3. 대응 플로우

1. 에러 확인
- 특정 API 엔드포인트에서 500 에러 발생 확인 (서비스 내 도서 검색 API)

2. 재현 확인
- 동일 엔드포인트 단건 요청 → 500 재현 여부 확인
- 부하와 무관하게 발생하는지 확인 → 일정 이상이 되면 500

3. 로그 확인
- Docker 로그에서 에러 메시지 확인 → timeout 에러 다수 확인

4. API 단독 테스트
- API 단독 호출 테스트
- 응답 지연 및 간헐적 실패 확인

5. 병목 구간 특정
- 해당 API 호출 구간에서 latency 급증 확인

6. 코드 확인
- 백엔드 코드 로직에서에서 문제 가능성 확인
- 외부 API(Google Books API) 호출에 대한 재시도/제한 처리가 없음 확인


### 4. 초기 대응

- 기존에 있던 Redis 캐시 레이어에 동일 검색어에 대한 검색 결과 캐싱 로직 개발
- 외부 API 호출에 timeout 설정 추가 → 불필요한 대기 요청 차단, 적체 완화

### 5. 원인 분석

- 원인: 검색 결과 API는 내부적으로 Google Books API를 호출함. API 제공자 측의 Rate Limit 및 timeout 제한으로 인해, 요청이 적체되며 에러율 급증.
- 서비스가 의존하고 있는 외부 API의 제한 및 정책을 정확히 파악하지 대응 계획을 세워두지 않아 발생한 문제.

### 6. 사후 방지 대책 🚀

- 외부 의존성 관련 Rate Limit, timeout 정책 사전에 파악하고 대응 전략 수립 (캐시 및 fallback 전략 등)
- 외부 API 호출 구간 로그 및 메트릭 강화
- 에러율, latency 기반 알림 시스템 도입
