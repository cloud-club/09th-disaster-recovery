## A. 🧱 기초 개념

### 1) SLO / SLI / SLA란?

> 서비스 제공자가 서비스 이용자에게 서비스를 제공함에 있어 고려하는 각 지표/단계 별 용어

- SLI: Service Level Indicator; 서비스의 수준을 판단하기 위한 실제 측정치
- SLO: Service Level Objective; SLI를 바탕으로 설정한 내부적 목표치로서, 엔지니어링 팀이 달성해야 할 실질적 기준
- SLA: Service Level Agreement; 서비스 제공자와 고객 간 법적계약내용으로, SLO보다 조금 더 여유를 두고 기준을 설정


### 2) Metrics / Logs / Traces란?

> Observability의 3요소; 복잡한 시스템의 상태를 파악하고 문제를 진단하는 데 필수적인 데이터

- Metrics: 일정한 시간 간격으로 수집되는 특정 환경에 대한 시계열 데이터
- Logs: 시스템 내에서 발생하는 개별 이벤트에 대한 상세 기록
- Traces: 요청이 시스템 내부를 어떻게 흘러갔는지 추적한 데이터

## B. 각자의 생각

### 장애를 어떻게 대응해야 할까? / 장애 대응에서 가장 중요한 것은 무엇일까?

개인적으로는 '되게 만드는 것'이 가장 중요하다고 생각합니다.
서비스는 비즈니스적 가치를 담기 위해 존재하고, 이는 결국 니즈를 가진 수요자에게 도달했을 때 그 목적을 이루기 때문입니다.

하지만 이 가치를 전달할 수 없게 된다면 서비스의 필요성에 대한 문제가 발생하므로,
장애가 발생한다면 우선 **동작하는 상태**로 다시 만들고, 이후 문제에 대한 원인분석 및 추후 대응을 하는 것이 옳다고 생각합니다.

## C. Runbook 작성

> 본 시나리오는 작년 하반기 발생했던 교내 클라우드 인프라 실제 장애 대응과정에서의 상황을 바탕으로 구성하였습니다.

### 1. 장애 시나리오

- 테스트베드 구성을 위해 유휴서버에 대한 초기화작업을 진행하였다.
- 직후 운영환경 인스턴스에 접근되지 않는다는 장애접수가 들어오기 시작하였다.

**#환경**
- HW RAID System 기반 2-copy RAID10 디스크 베이스
- MicroCeph 기반 3노드, 2-copy HA 디스크 환경
- Openstack 2025.1 Cluster (Control-Compute Splition)
- 로그 모니터링 및 알림환경 부재 (Openstack 기본 모니터링 설정은 완료)

**#사용자_영향**
- 일부 인스턴스 접근 불가
- 데이터조회 시 일부 디렉토리에 대한 접근 불가

### 2. 장애 인지
- Ceph Cluster CLI 경고: 정합성 손실 및 정족수 미달 경구
- 사용자 장애접수: 인스턴스 접근장애 및 데이터 접근오류 관련 보고


### 3. 대응 플로우

1. Openstack Admin Dashboard 내 시스템로그 확인
  - Error Log
  - System Log
  - API Call Log

2. Openstack 디스크 서비스 (Cinder) 상태 확인
  - cinder-api 컴포넌트 상태 및 로그 확인 (/var/log/kolla/cinder/cinder-api.log)
  - cinder-volume 컴포넌트 상태 및 로그 확인 (/var/log/kolla/cinder/cinder-volume.log)
  - cinder-scheduler 컴포넌트 상태 및 로그 확인 (/var/log/kolla/cinder/cinder-scheduler.log)

3. Ceph Cluster 상태 확인
  - 상태확인 (microceph.ceph health detail)
  - OSD 트리 확인 (microceph.ceph osd tree)

4. HW RAID 상태 확인
  - RAID Controller 상태 확인
  - 디스크 배드섹터 체크

### 4. 초기 대응

- 정족수 미달 해결을 위한 비활성화 노드에서의 서비스 활성화
- 전체 Microceph 비활성화 및 단독 Quarum 구성, 정합성 보정
- Monmap 백업

### 5. 원인 분석

- 장애원인은 해당일 작업했던 '유휴서버'가 운영환경의 Ceph 클러스터에 포함되어있었다는 사실이었습니다.
- Microceph 내부에서 모종의 사유로 '유휴서버'에 대한 클러스터 포함여부가 표시되지 않았고, 이로 인해 cli환경에서 클러스터 조회 시 노드의 클러스터 포함여부가 표시되지 않았습니다.
- 정족수는 최소 3이기 때문에 노드가 1개 손실이 난 순간 서비스가 모두 정지되었고, 특히 해당 노드에'만' 저장되어있던 사용자 VM의 인스턴스는 즉시 장애가 발생하게 되었습니다.

### 6. 사후 방지 대책

- 기반 디스트 HA시스템 교체 (Microceph -> Ceph)
  - Microceph은 Ceph에 비해 자료가 많지 않아, 장애대응 간 레퍼런스 탐색에 어려움을 겪었습니다.
  - 또한 Web Dashboard도 없어서, 정상적인 모니터링 / 제어가 어려웠습니다.
- 모니터링 인프라 구축
  - 사용자의 장애접수 전까지 장애의 발생사실 조차 알지 못했습니다.
  - 또한 각 컴포넌트의 로그를 찾기 위해 서버 전체에 대한 검색을 수행하는 등 불필요한 부차업무가 증가하였습니다.
- 디스크 클러스터 보완
  - 실제 물리서버의 수량을 증가시켜 2대까지 장애가 발생하더라도 견딜 수 있도록 노드를 추가했습니다.
- 클러스터 손실 핫픽스북 작성
  - 장애상황에 대한 상황 별 시나리오를 테스트하고, 각 상황 별 긴급대응가이드를 작성하여 공유했습니다.
