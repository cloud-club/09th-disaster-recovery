## **기초 개념**

</aside>

### `SLI (Service Level Indicator)`

- **서비스의 성능을 측정하는 실제 데이터**
- 특정 서비스의 품질을 측정하기 위한 핵심 지표
- 사용자 경험과 직접적으로 관련이 있음
- ex. 웹사이트 응답 속도, 서비스 가용성, 오류율

### `SLO (Service Level Objective)`

- **SLI를 기준으로 설정한 목표 값**
- 서비스의 성능을 일정 수준 이상으로 유지하기 위해 정한 목표값
- SLO를 통해 서비스의 신뢰성을 유지하거나 개선할 수 있음
- ex. 1달 동안의 가용성은 99.95% 이상이어야 한다

### `SLA (Service Level Agreement)`

- **SLO를 기반으로 고객과 계약한 서비스 수준으로 미달시 보상이 따름**
- 서비스 제공자와 고객 간의 공식적인 계약
- ex. 가용성 99.95% 미달시 환불

### `Logs`

- **시스템에서 요청이 처리되는 과정 중 특정 시점에 일어난 이벤트에 대한 기록**
- 특정 시점에 발생한 이벤트의 맥락과 원인 분석을 위해 사용

### `Metrics`

- **숫자 표현으로 시스템을 전체적으로 모니터링하는데에 도움을 줌**
- 시스템의 이상 징후를 감지하고, 시간 대별 동작 분석을 통해 최근의 추세를 알아내는데 사용

### `Traces`

- **하나의 요청이 시작부터 끝날때까지 시스템 여러 구성요소를 어떻게 지나가는지에 대한 흐름**
- 전체 경로 파악 및, 병목 지점 발견을 위해 사용

## **각자의 생각**

</aside>

### `장애를 어떻게 대응 해야할까?`

장애는 신속하게 대응해야 한다고 생각합니다. 

장애 상황에는 분석보다 복구가 최우선이 되어야 합니다. 따라서, 원인을 파악하느라 복구 골든 타임을 놓치지 않는 태도가 중요하다고 생각합니다. 장애에 대한 근본적인 원인을 파악하지 못했다면, 즉각적인 롤백이나 트래픽 우회 등을 통해 서비스의 가용성을 우선적으로 확보하는것이 중요하다고 생각합니다.

### `장애 대응에서 가장 중요한 것은 무엇일까?`

저는 장애 대응에서 가장 중요한 것은 ‘얼마나 빠르게 장애를 인지하는가’라고 생각합니다.

물론 장애를 사전에 예방하는 것과, 복구 체계를 잘 갖추는 것 모두 중요합니다. 하지만 아무리 철저하게 대비하더라도 모든 장애를 완전히 막을 수는 없고, 잘 설계된 복구 체계 역시 장애를 인지한 이후에야 비로소 의미를 갖는다고 생각합니다. 즉, 인지는 장애 대응 프로세스의 전제 조건이자 트리거이기 때문에 가장 중요도가 높다고 생각합니다.

또한 장애는 시간이 지날수록 영향 범위가 커지기 때문에, 얼마나 빨리 인지했는지가 전체 피해 규모를 결정짓는 핵심 요소가 됩니다. 초기에는 작은 이상으로 끝날 수 있는 문제가, 인지가 늦어지면 서비스 전반의 장애로 확산될 수 있기 때문입니다.

그래서 저는 장애 대응에서 가장 중요한 것은 복구 속도 이전에, 얼마나 빠르게 이상 징후를 감지하고 문제를 인지할 수 있는가라고 생각합니다.

## **장애 대응 플로우 런북**

</aside>

### `장애 시나리오`

- 배포를 위해 Jenkins에서 브랜치 스캔을 수행하던 중, 일정 시간이 지난 후 Jenkins 접근 시 502 Bad Gateway가 발생한 상황

### `장애 인지`

- Jenkins UI 접속 시 502 Bad Gateway 발생
- Grafana에서 CPU 사용률 급증 확인
- 빌드/스캔 작업이 비정상적으로 오래 지속됨

### `대응 플로우`

- 현재 장애 영향 범위 및 시스템 상태 확인 (CPU, Memory, 실행 중 Job)
- 과부하의 직접 원인(Job/프로세스) 식별
- 원인 작업 중단을 통한 부하 제거
- Jenkins 응답 상태 복구 (재기동 여부 판단 및 수행)
- 정상 동작 여부 확인 후 추가 조치 진행
- 원인 분석 및 재발 방지 대책 수립

### `초기 대응`

- 다른 Slave Job 영향 여부 확인 및 정리
- 과도한 리소스를 사용하는 브랜치 스캔 Job 식별
- 해당 Job 강제 종료 (kill)
- Jenkins 프로세스 재기동

### `원인 분석`

- 브랜치 스캔 시 브랜치 컨벤션 및 regex 기반 필터 없이 스캔이 진행되어 모든 레포지토리의 브랜치가 탐색 대상에 포함됨
- 과도한 리소스 사용으로 OOM이 발생하며 Jenkins 서비스가 다운됨

### `사후 방지 대책`

- 브랜치 스캔 범위를 제어하는 정책 (브랜치 컨벤션, regex 패턴 필터) 적용
