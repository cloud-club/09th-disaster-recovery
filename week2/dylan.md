## 1. 개념 학습

### SLO / SLI / SLA — 이론적 배경과 상세 설명

#### 📚 출처: Google SRE Book

SLO/SLI/SLA 개념은 **Google의 SRE(Site Reliability Engineering)** 체계에서 정립되었다.
Google이 약 20년간 대규모 서비스를 운영하며 쌓은 경험을 2016년에 책으로 공개했고, 이것이 업계 표준이 되었다.

- **Site Reliability Engineering** (2016) — Chapter 4: Service Level Objectives
- **The Site Reliability Workbook** (2018) — Chapter 2: Implementing SLOs
- 두 권 모두 [sre.google](https://sre.google) 에서 무료 열람 가능

> SRE의 핵심 아이디어는 **메트릭이 비즈니스 목표와 밀접하게 연결되어야 한다**는 것이다.
> SLO, SLA, SLI 없이는 시스템이 신뢰할 수 있는지, 가용한지, 유용한지조차 알 수 없다.

---

#### 한눈에 보는 요약표

| 구분 | 정의 | 예시 |
|------|------|------|
| **SLI** (Service Level Indicator) | 서비스 수준을 나타내는 **실제 측정값** | "지난 30일간 성공 응답 비율 = 99.97%" |
| **SLO** (Service Level Objective) | SLI에 대해 설정하는 **내부 목표치** | "월간 가용성 99.95% 유지" |
| **SLA** (Service Level Agreement) | SLO를 포함하는 **고객과의 계약**, 못 지키면 보상 | "월간 가용성 99.9% 미달 시 크레딧 환불" |

> 💡 SLI(측정) → SLO(목표) → SLA(계약) 순서로 흐른다.
> 측정값이 목표에 못 미치면 계약 위반이 되는 구조.

---

#### SLI (Service Level Indicator) — "어떻게 측정할 것인가"

SLI는 서비스가 얼마나 잘 동작하고 있는지를 나타내는 정량적 지표다.
Google SRE Book에서는 SLI를 **좋은 이벤트 수 / 전체 이벤트 수** 비율로 정의할 것을 권장한다.

**SLI 공식 예시:**

```
SLI = 성공한 HTTP 요청 수 / 전체 HTTP 요청 수
SLI = 100ms 이내 완료된 gRPC 호출 수 / 전체 gRPC 호출 수
```

이 비율 형태를 쓰면 SLI가 항상 0%~100% 범위에 들어오기 때문에 직관적이고, 에러 예산 계산에도 바로 활용할 수 있다.

**대표적인 SLI 유형:**

| SLI 유형 | 측정 내용 | 적합한 서비스 |
|----------|-----------|--------------|
| **가용성(Availability)** | 성공 응답 비율 | 거의 모든 서비스 |
| **지연시간(Latency)** | 특정 임계치 이내 응답 비율 (p50, p99 등) | API, 웹 서비스 |
| **처리량(Throughput)** | 단위 시간당 처리된 요청 수 | 배치 시스템, 파이프라인 |
| **정확성(Correctness)** | 올바른 결과를 반환한 비율 | 데이터 처리, 검색 |
| **내구성(Durability)** | 데이터가 장기간 유지되는 확률 | 스토리지, DB |

> ⚠️ **평균(average)의 함정**: SRE Book에서는 평균값이 롱테일을 숨길 수 있으므로, p99 같은 **고 백분위수(high percentile)** 를 SLI로 사용할 것을 권장한다. 연구에 따르면 사용자는 응답이 약간 느린 시스템보다 응답 시간 편차가 큰 시스템을 더 싫어한다.

---

#### SLO (Service Level Objective) — "목표를 얼마로 잡을 것인가"

SLO는 SLI에 대한 **목표 범위** 다. "이 정도는 되어야 한다"는 내부 기준선이다.

**SLO 설정 예시:**

| SLO 종류 | 목표 | 의미 |
|----------|------|------|
| 가용성 SLO | 월간 99.95% | 4주간 허용 다운타임 ≈ 21.6분 |
| 지연시간 SLO | 요청의 99%가 200ms 이내 | p99 레이턴시 기준 |
| 에러율 SLO | 5xx 에러율 0.1% 이하 | 1,000건 중 1건 미만 |

**SLO 설정 원칙 (SRE Book 기반):**

1. **현재 성능을 그대로 SLO로 삼지 마라** — 현 성능에 묶이면 개선 여지가 사라진다. 단, 첫 SLO를 잡을 정보가 없다면 현재 성능에서 출발하되 반드시 반복적으로 개선하라.
2. **100%를 목표로 잡지 마라** — 100% 가용성은 비현실적이며, 비용이 기하급수적으로 증가한다. SLO가 100% 미만이라는 것을 인정하는 순간, 속도(feature velocity)와 안정성(reliability) 사이의 트레이드오프를 논의할 수 있게 된다.
3. **SLO는 비즈니스 의사결정에 활용되어야 한다** — 인용되지 않는 SLO는 만들 가치가 없다. SLO를 근거로 업무 우선순위를 바꿀 수 있어야 한다.
4. **첫 SLO가 완벽할 필요 없다** — 가장 중요한 것은 무언가를 측정하기 시작하고, 피드백 루프를 만드는 것이다.

**Nines(나인) 표기법:**

| 표기 | 가용성 | 월간 허용 다운타임 |
|------|--------|-------------------|
| 2 nines | 99% | 약 7.3시간 |
| 3 nines | 99.9% | 약 43.8분 |
| 3.5 nines | 99.95% | 약 21.9분 |
| 4 nines | 99.99% | 약 4.4분 |
| 5 nines | 99.999% | 약 26초 |

> Nine이 하나 늘 때마다 운영 비용과 복잡도가 급격히 올라간다.
> 대부분의 서비스는 3~4 nines가 적절하다.

#### SLA (Service Level Agreement) — "못 지키면 어떤 대가를 치르는가"

SLA는 SLO를 포함하는 **고객과의 공식 계약**이다. SLO와의 핵심 차이는 **위반 시 결과(보상/페널티)가 명시**되어 있다는 점이다.

**SLO vs SLA 구분법 (SRE Book):**
> "SLO를 못 지켰을 때 어떤 일이 일어나는가?"를 물어보라.
> 명시적 결과가 없으면 SLO이고, 금전적 보상 등 결과가 있으면 SLA다.

**실제 클라우드 SLA 예시:**

| 서비스 | SLA 가용성 | 조건 | 미달 시 최대 보상 |
|--------|-----------|------|------------------|
| AWS EC2 | 99.99% (Region-Level) | 멀티 AZ 배포 필수 | 최대 30% 크레딧 |
| AWS EC2 | 99.5% (Instance-Level) | 단일 인스턴스 | 최대 30% 크레딧 |
| Google Compute Engine | 99.99% (멀티존) | 2개 이상 Zone에 배포 | 최대 50% 크레딧 |
| Azure VMs | 99.99% | 가용성 영역 2개 이상 배포 | 최대 100% 크레딧 |

> ⚠️ SLA 수치와 보상 조건은 수시로 변경된다. 반드시 각 클라우드의 공식 SLA 페이지에서 최신 내용을 확인할 것.

**SLA ⊃ SLO ⊃ SLI 관계:**

SLA, SLO, SLI는 바깥에서 안쪽으로 포함 관계를 이룬다.

| 계층 | 내용 | 예시 |
|------|------|------|
| **SLA** (가장 바깥) | 고객과의 계약 + 위반 시 보상 | "월간 가용성 99.9% 미달 시 10% 크레딧 환불" |
| ↳ **SLO** (중간) | 내부 목표치 (SLA보다 빡빡하게) | "내부 목표: 월간 가용성 99.95%" |
| 　↳ **SLI** (가장 안쪽) | 실제 측정값 | "현재 측정값: 성공 요청 / 전체 요청 = 99.97%" |

즉, **SLI로 측정**하고 → **SLO와 비교**하고 → **SLA 위반 여부를 판단**하는 흐름이다.

> 💡 SRE 팀은 보통 SLA 작성에 직접 관여하지 않는다 (비즈니스/법무 영역).
> 대신 SLI를 정의하고, SLO 위반을 방지하는 데 집중한다.
> Google 검색처럼 공개 SLA가 없는 서비스도 내부적으로는 SLI와 SLO를 반드시 관리한다.

### 관측 3종 세트 — Metrics / Logs / Traces

서비스 관측(Observability)의 세 기둥이라 불리는 요소들이다.

#### 1) Metrics (메트릭)

> **"지금 시스템이 어떤 상태야?"** 에 대한 숫자 요약

- 시계열 숫자 데이터 (CPU 사용률, 요청 수, 에러율 등)
- 저장 비용이 낮고 집계·알람에 최적화
- 대표 도구: **Prometheus**, CloudWatch, Datadog

```
# Prometheus 메트릭 예시
http_requests_total{method="GET", status="200"} 12345
http_request_duration_seconds_bucket{le="0.5"} 9800
```

**핵심 메트릭 — RED / USE 방법론**

| 방법론 | 대상 | 항목 |
|--------|------|------|
| **RED** | 서비스(요청 기반) | **R**ate(요청률), **E**rror(에러율), **D**uration(응답시간) |
| **USE** | 리소스(인프라) | **U**tilization(사용률), **S**aturation(포화도), **E**rrors(에러) |

#### 2) Logs (로그)

> **"그때 무슨 일이 있었어?"** 에 대한 상세 기록

- 이벤트 단위의 텍스트/구조화 데이터
- 디버깅의 핵심 — 에러 스택트레이스, 요청 파라미터 등
- 대표 도구: **Loki**, ELK(Elasticsearch + Logstash + Kibana), CloudWatch Logs

```json
{
  "timestamp": "2025-03-15T09:12:34Z",
  "level": "ERROR",
  "service": "payment-api",
  "message": "결제 처리 실패",
  "error": "TimeoutException: upstream 응답 없음",
  "trace_id": "abc123def456"
}
```

#### 3) Traces (트레이스)

> **"이 요청이 어디를 거쳐서 느려졌어?"** 에 대한 흐름 추적

- 분산 시스템에서 하나의 요청이 여러 서비스를 거치는 경로를 시각화
- 각 구간(Span)의 소요 시간을 파악 → 병목 지점 발견
- 대표 도구: **Jaeger**, Zipkin, Tempo, AWS X-Ray

**Trace 흐름 예시 — 하나의 요청이 서비스를 거치는 과정:**

| 순서 | 서비스 | 소요 시간 | 비고 |
|------|--------|----------|------|
| 1 | Client | - | 요청 시작 |
| 2 | API Gateway | 5ms | 정상 |
| 3 | Auth Service | 12ms | 정상 |
| 4 | **Payment Service** | **340ms** | ⚠️ 병목 지점 |
| 5 | DB | 15ms | 정상 |

→ 이 Trace를 통해 Payment Service가 병목임을 즉시 파악할 수 있다.

#### 세 가지가 만나는 지점

| 상황 | 활용 흐름 |
|------|-----------|
| 알람 발생 | **Metrics**에서 이상 감지 (에러율 급등) |
| 원인 파악 | **Logs**에서 해당 시간대 에러 로그 확인 |
| 범위 추적 | **Traces**로 어느 서비스에서 실패했는지 추적 |

> 🔑 `trace_id`로 세 가지를 연결하는 것이 핵심이다.

## 2. Loki + Grafana로 로그 수집/검색/대시보드 구축

### 왜 Loki인가?

Elasticsearch(ELK)는 강력하지만 운영 부담과 비용이 크다. Loki는 **로그 본문을 인덱싱하지 않고 라벨만 인덱싱**하는 전략으로 비용과 복잡도를 대폭 낮췄다.

| 비교 | ELK Stack | Loki |
|------|-----------|------|
| 인덱싱 | 로그 전체 풀텍스트 인덱싱 | 라벨(metadata)만 인덱싱 |
| 스토리지 비용 | 높음 | 낮음 (S3/GCS 활용 가능) |
| 쿼리 속도 | 빠름 (인덱스 덕분) | 라벨 필터 후 grep 방식 |
| 운영 난이도 | 높음 (JVM 튜닝 등) | 낮음 |
| Grafana 연동 | 별도 설정 필요 | 네이티브 통합 |

### 구성도 (이론적)

Loki + Grafana 로그 파이프라인은 크게 3단계로 구성된다.

| 단계 | 구성 요소 | K8s 리소스 | 역할 |
|------|----------|-----------|------|
| **1. 로그 생성** | App Pod 1, 2, … N | Deployment / StatefulSet | 각 Pod가 stdout으로 로그 출력 |
| **2. 로그 수집** | **Promtail** | DaemonSet (노드마다 1개) | 각 노드의 Pod 로그를 수집하고 라벨(namespace, pod, app 등)을 부착하여 Loki로 전송 |
| **3. 로그 저장** | **Loki** | StatefulSet | 라벨만 인덱싱하고 로그 본문은 청크(chunk)로 압축 저장. S3/GCS를 백엔드 스토리지로 활용 가능 |
| **4. 시각화** | **Grafana** | Deployment | Loki를 데이터소스로 연결하여 LogQL 쿼리, 대시보드, 알람 설정 |

> 💡 데이터 흐름: **App Pod → (stdout) → Promtail → Loki → Grafana**

### LogQL 기본 쿼리 예시

```logql
# 특정 앱의 에러 로그만 보기
{app="payment-api"} |= "ERROR"

# JSON 로그에서 status_code 필터링
{namespace="production"} | json | status_code >= 500

# 최근 5분간 에러 발생률 (메트릭 쿼리)
rate({app="payment-api"} |= "ERROR" [5m])
```

### Grafana 대시보드 구성 팁

1. **로그 볼륨 패널** — `rate()` 쿼리로 시간대별 로그 양 시각화
2. **에러 로그 테이블** — `|= "ERROR"` 필터로 에러만 추출해 테이블로 표시
3. **서비스별 로그 레벨 비율** — `sum by (level)` 집계로 파이 차트 구성
4. **변수(Variable) 활용** — `namespace`, `app` 등을 드롭다운으로 만들어 동적 필터링

### 다른 도구를 쓰고 있다면?

어떤 도구든 핵심 원리는 같다.

- **CloudWatch Logs** — AWS 네이티브, 별도 에이전트 불필요하지만 쿼리 비용 주의
- **ELK Stack** — 풀텍스트 검색이 강력, 대규모 운영 시 클러스터 관리 필요
- **Datadog Logs** — SaaS형, 빠른 도입 가능하지만 볼륨 기반 과금

> 🙌 현직에서 사용 중인 도구와 경험이 있다면 자유롭게 공유해 주세요!

## 3. 각자의 생각 (자유롭게)

### 마인드셋

완벽한 시스템은 없다. 장애는 항상 기상천외하게 터진다. 한번 겪어보면 깨닫는 건, **"뭘 기록하고 있었느냐"**가 대응 속도를 결정한다는 것이다. 터진 후에 "로그가 없네", "메트릭을 안 쌓고 있었네" 하면 이미 늦다. SLI/SLO, Metrics/Logs/Traces를 **장애 전에** 알고 세팅해두자. 그게 이번 스터디를 하는 이유다.

결국 MTTR(복구 시간)을 줄이는 가장 확실한 방법은 MTTD(인지 시간)를 줄이는 것이다. 복구에 30분이 걸려도 1분 만에 인지하면 피해를 최소화할 수 있지만, 5분이면 끝날 장애도 인지에 20분 걸리면 사용자는 25분을 고통받는다.

|구분|의미|결과|
|---|---|---|
|**False Positive**|장애 아닌데 알람 울림|반복되면 Alert Fatigue → 진짜 알람도 무시|
|**False Negative**|장애인데 알람 안 울림|CS 전화로 장애 인지 → 모니터링 무의미|

대부분의 팀이 False Negative를 두려워해서 알람을 잔뜩 건다. 그런데 정작 팀을 망가뜨리는 건 **False Positive** 쪽이다. 거짓 알람이 쌓이면 "또 저거겠지" 하고 무시하는 문화가 생기고, 그 순간 진짜 장애 알람도 묻힌다 — 결국 False Positive가 False Negative를 만드는 셈이다.

CPU 80% 같은 인프라 고정 임계치로는 이걸 해결할 수 없다. **SLI/SLO 기반 알람**이 이 트레이드오프를 개선한다 — 사용자 경험을 직접 측정하니까.

또한, 보지 못하는 곳에서는 알람 자체가 울릴 수 없다. **"어디를 모니터링하고 있는가"보다 "어디를 안 보고 있는가"를 먼저 점검하자.**

# 🚨 런북: C사 ETL 파이프라인 장애 대응기

## 배경

C사는 이커머스 셀러들의 판매·광고 데이터를 50+ 채널에서 크롤링해 대시보드로 보여주는 B2B 서비스를 운영하는 스타트업이었다. 크롤링 파이프라인은 Prefect + Kubernetes 기반이었고, 헤드레스 브라우저를 돌려서 채널별 데이터를 긁어오는 구조. 전체 파이프라인이 다 도는 데 5~6시간이 걸렸다.

문제는, **스타트업이라 돈이 없었다.** Prefect도 프리플랜으로 쓰고 있었고, 별도 로깅 시스템 같은 건 구축하지 않은 상태였다. 참고로 나는 입사한 지 얼마 안 된 상태였다. 이 구조를 만든 건 내가 아니다. (중요)

---

## 장애 발생 — "출근했는데 데이터가 없다"

출근 전에 전화가 왔다. 파이프라인이 안 돌고 있다고. 고객이 보는 대시보드에 최신 데이터가 없으니, 이건 바로 CS 문의로 이어지는 상황이었다.
근데 뭐가 문제인지를 모르겠었다. 간헐적으로 실패하는 수준이 아니라, 거의 계속 대기 상태에 빠져 있었다. 뭘 봐야 하는지조차 감이 안 잡혔다.

---

## 원인 파악 — "로그가 없다"

원인을 찾으려고 했는데, **볼 게 없었다.** Prefect 로그에는 "Job 실패"만 찍혀 있고 왜 실패했는지는 안 나왔다. K8s Pod 로그를 보려고 해도 단서가 없었다. OOM(Out of Memory)은 리눅스 커널이 SIGKILL로 프로세스를 즉사시키기 때문에, 애플리케이션이 "나 지금 메모리 부족으로 죽는다"는 로그를 남길 기회 자체가 없다. K8s 이벤트에는 `OOMKilled`라는 기록이 남긴 하지만, K8s 이벤트의 기본 보존 시간은 약 1시간이다. 파이프라인은 새벽에 돌았고, 아침에 출근했을 때는 이미 이벤트가 삭제된 뒤였다. 흔적이 없으니 뭐가 문제인지 알 수가 없었다.

한참을 헤매다가, 아직 살아있는 Pod에서 `kubectl describe pod`으로 상태를 까보니 `Last State: OOMKilled`가 찍혀 있었다. 여기서 퍼즐이 맞춰졌다.

**특정 채널의 Job이 메모리 초과로 OOMKilled → 근데 restartPolicy 때문에 무한 재시도 → 좀비 Job이 클러스터 리소스를 계속 잡아먹음 → 다른 정상 Job들까지 리소스 부족으로 실패 → 전체 파이프라인 중단**

하나의 채널이 죽으면서 전체를 끌고 내려가는 구조였던 거다.

---

## 초기 대응

원인을 파악한 후에는 빠르게 움직였다.

1. 문제 Job의 `restartPolicy`를 수정해서 무한 재생성을 즉시 차단
2. 좀비 Job 수동 삭제로 클러스터 리소스 확보
3. 나머지 정상 채널 파이프라인 수동 재실행
4. 실패한 채널은 메모리 limit을 올린 뒤 별도로 수동 실행

---

## 이후 개선 — "다시 터지면 이번엔 바로 알아야 한다"

초기 대응 이후, 이대로 두면 또 똑같이 당한다고 판단했다. 두 가지를 했다.

**1) Docker 메모리 최적화 및 limit 조정**

cgroup 메모리 사용량을 계측해서 컨테이너별 memory request/limit을 재산정했다. 헤드레스 브라우저가 이미지까지 렌더링하면서 메모리를 과도하게 잡아먹고 있었는데, 이미지 렌더링을 비활성화해서 메모리 사용량을 줄였다.

**2) Loki + Grafana + Kubernetes Event Exporter 도입**

이게 핵심이었다. 이번 장애에서 가장 시간을 잡아먹은 건 복구가 아니라 **"뭐가 문제인지 모르는 시간"**이었다. Prefect 로그로는 K8s 레벨의 OOM 같은 건 보이지 않았다. K8s 이벤트에 OOMKilled 기록이 남긴 하지만, 기본 보존이 1시간이라 새벽에 터지면 아침에는 이미 사라진 뒤다. 게다가 Promtail(로그 수집 에이전트)만으로는 Pod의 stdout/stderr만 수집하지, K8s 이벤트는 수집하지 않는다. 그래서 Kubernetes Event Exporter를 별도로 설치해서 K8s 이벤트를 Loki로 내보내도록 구성하고, Grafana에서 이상 징후 시 이메일 알림이 오도록 설정했다.

---

## 결과

- OOM 기반 장애: 월 20+회 → 0회
- 장애 인지 방식: "출근해서 눈으로 확인" → 알림 기반 즉시 인지

## 돌아보며

이 경험에서 깨달은 건 하나다. **로그가 없으면 장애 "대응"이 아니라 장애 "수색"을 하게 된다.** 뭐가 문제인지 찾는 데 대부분의 시간을 쓰고, 정작 고치는 건 금방이었다. 스타트업이라 돈이 없어서 로깅을 안 했는데, 돌이켜보면 로깅이야말로 가장 먼저 했어야 할 투자였다.

## 런북: ETL 파이프라인 장애 시 대응 플로우

### 알람 수신 (Grafana 이메일 알림)

1. Grafana 대시보드 접속 → 어떤 채널에서 실패했는지 확인
2. `kubectl get pods --field-selector=status.phase=Failed` 로 실패 Pod 확인
3. `kubectl describe pod <pod-name>` → OOMKilled 여부 확인

### OOMKilled 확인 시

1. **즉시 차단**: 해당 Job 삭제 (`kubectl delete job <job-name>`) → 좀비 Job의 무한 재시도로 클러스터 리소스 고갈되는 것 방지
2. **복구**: 나머지 정상 채널 파이프라인 정상 동작 확인 → 실패 채널은 memory limit 상향 후 수동 재실행
3. **사후**: Loki에서 해당 시간대 로그 확인, OOM 발생 원인 분석 (데이터 규모 증가? 크롤링 대상 변경?)

### OOMKilled가 아닌 경우

1. Loki에서 해당 채널 로그 조회: `{namespace="etl", channel="채널명"}`
2. 외부 의존성 문제인지 확인 (크롤링 대상 사이트 차단, API 변경 등)
3. 원인에 따라 코드 수정 또는 수동 재실행
